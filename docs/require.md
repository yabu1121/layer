## 1. プロダクト概要

### 1.1 3 段構造

```
ビジョン:    ひととのつながりで世界を明るく

コンセプト:  舞台はタイムラインから、世界へ。
             場所に積もる、友達との共感の地図

コア体験:    自分が惹かれた場所に想いを残すと、
             同じ場所で同じことを感じていた友達が見つかる
             —— 共通の思い出が地図の上に積み重なっていく
```

### 1.2 一言サマリ

**友達と、場所を介して、感性が重なる瞬間を見つける SNS。**

### 1.3 何が新しいか

- フォロー関係を主軸にしない（SNS だが、フィードがタイムラインではなく地図）
- 投稿は「個人の発信」ではなく「場所への寄稿」
- リアクションは「いいね」ではなく「わかる」（共感ベース）
- リアルタイム同期ではなく、**後から気付く重なり**が体験の核

### 1.4 既存サービスとの違い

| プロダクト | 軸 | 関係 | Layer との違い |
|---|---|---|---|
| Instagram | 投稿フィード | 友達 + フォロー | Layer は「誰の」より「どこの」 |
| BeReal | 同時刻の今 | 友達 | Layer は同期不要、後から共感が見つかる |
| Snapchat Map | リアルタイム位置 | 友達 | Layer は「想い」も残る |
| Locket Widget | ホーム画面写真 | 友達 | Layer は場所が主軸 |
| **Layer** | **場所への想い** | **友達中心** | **「同じ場所で同じこと感じた」を可視化** |

## 2. ターゲットユーザー

### 2.1 ペルソナ（メイン）

**山田リョウ・21 歳・大学 3 年**

- 都内私大、サークル・バイト・インターンで毎日違う場所に行く
- iPhone ヘビーユーザー、BeReal / Instagram / X 利用、ポケモン GO に再びハマっている
- 「新しい場所」「新しい人」は好き、でも自分から DM を送るのは苦手
- 趣味: 街歩き、カフェ巡り、ライブハウス
- SNS 疲れを感じている、フォロー関係が重い
- ゲーム要素・コレクション要素が好き、Strava の称号集めに反応する

### 2.2 ペルソナ（サブ）

**佐藤アヤ・19 歳・大学 1 年**

- 地方から上京、まだ友達は少ない
- 大学・サークル・バイト先で「同じ場所にいる人」と緩く繋がりたい
- マッチングアプリは抵抗あるが、友達は欲しい
- BeReal で同期と繋がっているが、もっと「街」で繋がりたい

### 2.3 共通する課題

- 既存 SNS は「すでに知ってる人」と繋がるためのもので、**関係の深化**が起こりにくい
- 場所に行っても「来た」記録が Instagram のストーリーに流れて消える
- 街そのものに価値を感じるが、それを残す・共有する手段がしっくり来ない

## 4. MVP スコープ

### 4.1 MVP の定義

> 2 週間で完成させる、「発見の体験」が成立する最小プロダクト。

### 4.2 成功シナリオ（ハッカソンデモ）

1. リョウがアプリを開いて、いる場所に Pin を立てる
2. リョウのアプリの地図に、友達のアヤが過去に同じ場所に立てた Pin が表示される
3. リョウが「同じ場所だ！」と気付き、アヤの Pin に「わかる」と共感を返す
4. アヤのアプリに通知が届き、「リョウと共通の場所」リストにこの場所が追加される

このシナリオが動けばコア体験 = **発見**が成立。

### 4.3 Must（MVP 必須）

US-A1, US-A2, US-A3, US-A4, US-A5, US-A9, US-B1, US-B2, US-C1, US-C2, US-D1, US-D2, US-D3, US-E1, US-E3 の 15 ストーリー。

### 4.4 Should（時間があれば）

US-B3 写真添付、US-B4 感情ラベル、US-C3 コメント、US-E2 地図操作の作り込み、US-E4 リスト、US-D4 地図上の共通場所可視化、US-A8 プロフィール編集。

### 4.5 Won't（MVP では作らない）

US-A6, US-A7, US-B5, US-B6, US-B7, US-C5, US-C6, US-D5, US-D6, US-E5, US-E6, US-F1〜US-F3。
プッシュ通知、画像アップロード、Pin の編集・削除、公開範囲設定、ブロック機能。

### 4.6 重視する非機能要素

**「発見」**。
クローズドな点、静かな UI、オープン要素より、**「その場で友達のコンテンツが見つかる」体験**を優先する。

## 5. 機能要求

各機能要求に「由来ストーリー」と「MVP 区分」を明記する。本セクションは原則すべて MVP 必須（Must）。Should / Won't は §4 を参照。

### FR-1. アカウント

由来: US-A1, US-A2, US-A8, US-A9

- FR-1.1 Google アカウントでサインイン … *MVP*
- FR-1.2 初回起動時にユーザー名・アイコン設定（アイコンは絵文字選択） … *MVP*
- FR-1.3 ユーザー ID（例: `riyo_1234`）が自動生成され、画面に表示される。オンボーディング画面で編集も可能 … *MVP*
- FR-1.4 プロフィール（名前・アイコン）を後から編集できる … *Should*
- FR-1.5 ログアウトできる … *MVP*

### FR-2. 友達

由来: US-A3, US-A4, US-A5

- FR-2.1 ユーザー ID で友達検索（完全一致のみ） … *MVP*
- FR-2.2 友達申請を送る … *MVP*
- FR-2.3 自分宛の申請を承認・拒否 … *MVP*
- FR-2.4 友達招待リンクの共有（ディープリンクで友達追加画面を開く） … *MVP*

### FR-3. Pin 投稿

由来: US-B1, US-B2

- FR-3.1 FAB（フローティングボタン）から投稿画面へ … *MVP*
- FR-3.2 現在地を自動取得し、地図上でピンを表示（位置の微調整可） … *MVP*
- FR-3.3 テキスト入力（最大 200 文字） … *MVP*
- FR-3.4 「投稿する」で確定、投稿後は地図に戻る … *MVP*

### FR-4. 地図

由来: US-C1, US-C4, US-E1, US-E2, US-E3

- FR-4.1 アプリ起動時のトップ画面として地図を表示 … *MVP*
- FR-4.2 自分の現在地を中心に、自分と友達の全 Pin を表示（RPC `get_visible_pins`、model.md §3.1） … *MVP*
- FR-4.3 自分の Pin と友達の Pin で色やアイコンを区別 … *MVP*
- FR-4.4 ピンをタップするとボトムシートで詳細を表示 … *MVP*

### FR-5. Pin 詳細

由来: US-C2, US-D1, US-D3, US-E3

- FR-5.1 Pin の本文、投稿者、投稿日時を表示 … *MVP*
- FR-5.2 同じ場所（半径 20m 以内）の他の Pin もリスト表示 **← 発見の核** … *MVP*
- FR-5.3 共感ボタン（「わかる」1 種類のみ） … *MVP*
- FR-5.4 共感した人の数とアイコンを表示 … *MVP*

### FR-6. 通知（in-app）

由来: US-D2, US-D3

- FR-6.1 アプリ起動時にトップに「お知らせ」バナーを表示 … *MVP*
- FR-6.2 自分の Pin に共感が来た時にバナー … *MVP*
- FR-6.3 友達が自分の過去 Pin と同じ場所に Pin を立てた時にバナー **← 発見の核** … *MVP*
- ※ プッシュ通知は MVP では実装しない。起動中のバッジ更新は Supabase Realtime で行う（発見・通知フローは screens.md §3.2）

## 6. 非機能要求

### 6.1 パフォーマンス

| 項目 | 目標 |
|---|---|
| 起動から地図表示まで | 3 秒以内 |
| Pin 投稿後、地図反映 | 1 秒以内 |
| 地図上の Pin 表示件数 | 100 件まではスムーズ |
| API レスポンス | p95 で 500ms 以内 |

### 6.2 プライバシー・セキュリティ

| 項目 | 方針 |
|---|---|
| Pin の公開範囲 | MVP では「友達のみ」固定 |
| 位置情報の精度 | 投稿時の正確な GPS を保存（友達には正確な位置が共有される点に留意） |
| 友達以外への露出 | ゼロ。DB レベルで保証する（Supabase RLS）。具体方針は model.md §5 を参照。`pins` の友達限定ポリシーには friendship 判定関数（`SECURITY DEFINER`）が必要で、付録 A の「最低限設定」だけでは不十分 |
| 認証 | Google OAuth、Supabase Auth が管理 |
| データ削除 | アカウント削除機能は MVP では未実装。**App Store 本番リリース前には必須**（Apple はアカウント作成機能を持つアプリに削除機能を要求）。TestFlight 配布のみなら審査対象外。§9.3 参照 |
| 通報・モデレーション | MVP では未実装。公開範囲が「友達のみ」固定のためリスクは限定的。公開機能（§9.2）導入時には必須。§9.3 参照 |

### 6.3 信頼性

| 項目 | 方針 |
|---|---|
| オフライン時 | アプリは起動、エラー画面で「ネット繋いでね」を表示 |
| 投稿失敗時 | エラートースト、リトライボタン |
| GPS 取得失敗 | 「位置情報を許可してください」のガイド |

### 6.4 ユーザビリティ

| 項目 | 方針 |
|---|---|
| 対応 OS | iOS 16+、Android 9+ |
| 言語 | 日本語のみ |
| アクセシビリティ | 文字サイズはシステム設定追従、コントラスト比 AA |
| オンボーディング | 3 画面以内で完了（Splash → SignIn → Onboarding） |

### 6.5 運用

| 項目 | 方針 |
|---|---|
| 配布 | TestFlight (iOS) / Internal Testing (Android) |
| ログ | Sentry で例外収集（時間あれば） |
| 分析 | MVP では入れない |

## 7. 技術スタック

### 7.1 クライアント

#### モバイル（`mobile/` — Flutter）

- **Flutter** 3.24+
- **状態管理**: Riverpod
- **ルーティング**: go_router
- **地図**: google_maps_flutter（API キーが必要）
- **地図クラスタリング**: google_maps_flutter 標準のクラスタリング（`ClusterManager` / `ClusterManagerId`。2.5+ で同一地点付近の Pin をまとめられる。旧 `google_maps_cluster_manager` は 2.14 系と非互換のため不使用）
- **位置情報**: geolocator
- **逆ジオコーディング**: geocoding（座標 → 場所名。`google_maps_flutter` 単体ではできない）
- **認証**: google_sign_in（取得した ID トークンを Go バックエンドで検証する）
- **API 通信**: dio（Go バックエンドの REST API を呼ぶ）
- **設定読み込み**: flutter_dotenv（`.env`）
- **その他**: flutter_svg, intl, shared_preferences, share_plus

#### Web フロントエンド（`frontend/` — Next.js）

- **Next.js**（App Router）/ **TypeScript**
- **スタイリング**: Tailwind CSS
- **API 通信**: fetch（Go バックエンドの REST API を呼ぶ）

### 7.2 バックエンド

- **DB**: Golang(gorm, echo, postgreSQL)
- **cloud**: Cloudflare

### 7.3 開発・配布

- バージョン管理: Git + GitHub

## 9. 将来の拡張

MVP リリース後、段階的に追加していく機能。

### 9.1 Phase 2: ゲーム要素

- 訪れた場所の「場所図鑑」コレクション
- 共感の「累計獲得数」「友達と重なった回数」などの実績
- 週次・月次のサマリ（Strava の Wrapped 風）

### 9.2 Phase 3: 公開機能

- 友達以外への公開 Pin（公開範囲を投稿時に選べる）
- 公開フィードでの発見
- ハッシュタグ・場所タグでの絞り込み

### 9.3 本番リリースに向けた必須対応

ハッカソン MVP（TestFlight / Internal Testing 配布）では不要だが、**App Store / Google Play への本番リリース前に必須**となる項目。

- **アカウント削除機能** — Apple はアカウント作成機能を持つアプリにアプリ内からの削除導線を要求する。`users` と関連レコードのカスケード削除を実装する。
- **通報・ブロック機能** — 公開機能（§9.2）を入れる場合は特に必須。US-A7（ブロック）の実装もここで行う。
- 利用規約・プライバシーポリシーの整備と同意フローの正式化（位置情報を扱うため要記載）。

### 9.4 Phase 4: 共感の表現拡張

- 「わかる」だけでなく「行きたい」「懐かしい」「守りたい」など複数感情ラベル
- Pin への感情ヒートマップ（場所ごとに集約表示）
- 「自分と感性が近い友達」リコメンド

### 9.5 Phase 5: ハードウェア拡張

ハッカソン後の長期構想:

- **物理ピン端末**: Pin を立てるための専用デバイス。NFC で場所と紐付け、街中に置けるオブジェクト
- **AR グラス表示**: 場所に立った Pin を空間に重ねて表示
- **街頭デバイス**: 公共スペースに置いて、その場所の Pin 集約を表示
- **コレクション物理化**: 訪れた場所がスタンプとして物理スタンプ帳に貯まる

### 9.6 Phase 6: その他のアイデア

- コメント機能（US-C3）
- Pin への写真添付（US-B3）
- グループ Pin（友達クラスタ単位で共有する Pin）
- 過去 Pin 近接通知（US-D6: 自分が以前 Pin を立てた場所に近づいたら通知）
- 時系列・場所タイプフィルタ（US-E5, US-E6）
